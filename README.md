# 🤖 الوكيل الذاتي التطور (Self-Evolving Agent)

وكيل برمجي يعمل داخل مستودع GitHub، يستخدم **Groq API** بنموذج **llama-3.3-70b-versatile** (70 مليار باراميتر، يدعم استدعاء الأدوات) لتحليل المستودع وتحسينه تدريجياً بشكل آمن — مصمم للتفكير النقدي قبل الفعل ومتابعة مشاكل طويلة المدى، لا فقط تنفيذ مهام يومية عشوائية.

## ⚙️ الإعداد

```bash
pip install -r requirements.txt
export GROQ_API_KEY="ضع_مفتاحك_هنا"
```

## 🚀 التشغيل

```bash
python main.py
# أو بتوجيه مخصص:
python main.py "ركّز هذه الدورة على مشكلة أداء أداة البحث"

# مراجعة معمارية أسبوعية (سياق أوسع، تعيد كتابة PLAN.md):
python scripts/architect_review.py

# تقرير أسبوعي من البيانات الفعلية:
python scripts/generate_weekly_report.py
```

## 🧠 معمارية التفكير: مرحلتان منفصلتان فعلياً، لا نداء واحد مباشر

هذا هو الفرق الجوهري عن نسخة "ينفذ فوراً": كل دورة تمر بمرحلتين، كل واحدة استدعاء API منفصل:

1. **مرحلة التفكير (`_deliberate` في `agent.py`)** — بدون أي أدوات إطلاقاً. النموذج يراجع
   `PLAN.md`، `PROBLEMS.md` (المشاكل المفتوحة وعدد محاولاتها السابقة)، وآخر الأحداث، ثم:
   يعدّد 2-4 مرشحين لأولوية هذه الدورة → ينتقد كل واحد (هل جُرّب وفشل؟ ما مخاطره؟) → يختار
   واحداً بسبب صريح → يكتب خطة تنفيذ من 3-6 خطوات محددة. **لا يلمس كوداً في هذه المرحلة.**
2. **مرحلة التنفيذ** — حلقة `tool-calling` عادية، لكنها مُقيَّدة بخطة المرحلة الأولى، لا ترتجل.

هذا يمنع النمط الشائع في الوكلاء البسيطة: "قرار وتنفيذ في نفس اللحظة" بلا مساحة تفكير حقيقية.

## 🗂️ سجل المشاكل طويلة المدى (`PROBLEMS.md` + `memory/problems.py`)

الفرق عن `PLAN.md` (مهام قصيرة قابلة للإنجاز في دورة أو أسبوع):
`PROBLEMS.md` يتابع مشاكل قد تحتاج أسابيع، مع **تاريخ كل محاولة سابقة وسبب فشلها** محفوظ في
`memory/problems.json`. قبل أي محاولة جديدة، يستدعي الوكيل `problem_op(action="history")`
ليرى ماذا جُرّب سابقاً ولماذا فشل — بدل تكرار نفس الخطأ كل أسبوع بلا وعي بالمحاولات الماضية.

## 🏛️ المراجعة المعمارية الأسبوعية (`scripts/architect_review.py`)

منفصلة تماماً عن الدورة اليومية: تقرأ سياقاً أوسع بكثير (آخر 100 حدث بدل 15، كل المشاكل
المفتوحة والمغلقة)، ولا تنفّذ أي كود — فقط تعيد كتابة `PLAN.md` بعد نقد صريح: هل الوكيل يدور
حول نفسه؟ هل مشكلة معينة عالقة بلا تقدم فعلي؟ هل هناك قرار معماري أكبر (إعادة هيكلة، حذف كود
ميت) لم يُقترح لأن الدورات اليومية مصممة لخطوات صغيرة فقط؟

## 🧰 الأدوات المتاحة في مرحلة التنفيذ

| الأداة | الوصف | الملف |
|---|---|---|
| `file_op` | قراءة / كتابة / حذف / سرد الملفات + git commit | `tools/file_tool.py` |
| `run_terminal` | تنفيذ أوامر shell (تثبيت مكتبات، تشغيل اختبارات...) | `tools/terminal_tool.py` |
| `search_code` | بحث نصي/regex محلي بالكامل — **بدون أي API خارجي** | `tools/search_tool.py` |
| `problem_op` | تسجيل/استرجاع تاريخ محاولات حل مشكلة طويلة المدى | `memory/problems.py` |

## 🛡️ طبقة الأمان

- **لا خروج عن حدود المستودع**، **مسارات محمية** لا تُعدَّل إلا بإذن صريح، **فحص صياغة قبل الحفظ**
  (`ast.parse` / `json.loads`)، **نسخ احتياطي تلقائي وتراجع**، **حظر أوامر طرفية خطيرة**، **مهلة زمنية**.
- **بوّابة اختبار حقيقية وليست اقتراحاً** (`safety/test_gate.py`): قبل أي `git commit`، تُشغَّل
  الاختبارات فعلياً (pytest إن وُجد، وإلا فحص صياغي على الملفات المعدّلة). إن فشلت، **يُرفض الـ commit
  برمجياً** بغض النظر عمّا "يقرره" النموذج — هذا مُختبَر فعلياً: كود بايثون معطوب جُرِّب إدخاله
  وتم رفضه قبل أن يدخل السجل.

> ⚠️ الوكيل يلتزم (commit) ويدفع (push) مباشرة **بدون Pull Request ومراجعة بشرية**، بناءً على
> طلبك. يُنصح بتجربته أولاً على مستودع تجريبي، ومراجعة `PLAN.md` و `PROBLEMS.md` و `reports/`
> دورياً، خصوصاً في البداية.

## 🧠 وعي الوكيل ببيئته (`environment.py`)

قبل كل دورة، يُبنى ملخص فوري: نظام التشغيل، إصدار بايثون، حالة `git status`، آخر 5 commits،
شجرة الملفات، وعينة من الحزم المثبتة.

## 🔁 التشغيل الآلي عبر GitHub Actions

| Workflow | التوقيت | الوظيفة |
|---|---|---|
| `self_improve.yml` | يومياً 03:00 UTC | دورة تفكير + تنفيذ عادية |
| `architect_review.yml` | كل اثنين 05:00 UTC | مراجعة معمارية، تعيد كتابة `PLAN.md` |
| `weekly_report.yml` | كل اثنين 06:00 UTC | تقرير حقيقي من `history.json` |

**لا تنسَ إضافة `GROQ_API_KEY` كـ Secret في إعدادات المستودع** (Settings → Secrets and variables → Actions).
